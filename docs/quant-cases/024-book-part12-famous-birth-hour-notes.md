# 第十二篇 名人時辰（261–275）

## 章旨
- 不可盲信名人公布時辰或坊間流傳八字
- 要用量化模型把候選時辰並列比較，再結合人生經歷、真太陽時與大運流年驗證

## 工程化做法

### App 內目前可用的做法
- 目前 BaZi Desktop 尚未提供「批次候選時辰比較」工具。
- 可用「直接輸入四柱」逐一輸入 2 到 3 個候選時辰，對照：
  - 總分
  - 強弱
  - 用神結論
  - 從格 / 假從
  - 大運與流年量化
- 若後續要做成批次工具，建議在 Rust 端以 `analyze_pillars` / `build_quant_model_response` 為核心，外層只負責組候選清單。

### 適合自動化的部分
- 四柱輸入與量化
- 候選時辰並列比較
- 章法差異標記：
  - 身強 / 身弱
  - 從 / 不從
  - 用神是否改變

### 不宜假裝自動化的部分
- 「哪個時辰一定正確」
- 因為本篇核心仍依賴：
  - 真太陽時
  - 命主實際人生經歷
  - 重大事件與歲運應驗

## 書上案例

### 趙薇
- 書傳四柱：
  - `丙辰 / 辛卯 / 癸亥 / 甲戌`
- 作者修正候選：
  - `丙辰 / 辛卯 / 癸亥 / 己未`
- 書中主張：
  - `甲戌時` 會把命局推向疑似從弱或弱極
  - `己未時` 則回到「正格身弱無疑」
- 目前程式：
  - `甲戌時`：`-148.9`
  - `己未時`：`-88.5`
- 結論：
  - 雖未達書中約 `-250` 的極端弱值
  - 但修正後確實明顯「較健康」，方向與書相符

### 歐巴馬
- 書傳時辰：
  - `辛丑 / 乙未 / 己午 / 癸酉`
- 過渡候選：
  - `辛丑 / 乙未 / 己巳 / 甲戌`
- 作者推定：
  - `辛丑 / 乙未 / 己巳 / 壬申`
- 書中數值：
  - `癸酉時`：總分約 `-117`
  - `壬申時`：總分約 `+30`
- 目前程式：
  - `癸酉時`：`+88.7`
  - `甲戌時`：`+83.1`
  - `壬申時`：`+29.3`
- 結論：
  - `壬申時` 的分數已接近書中 `+30`
  - 但書傳 `癸酉時` 目前仍是明顯 gap
  - 這個案例適合持續保留為「候選時辰 gap case」

### 郭晶晶之子
- 書中比較的是兩個時辰：
  - `癸巳 / 庚申 / 乙丑 / 己卯`
  - `癸巳 / 庚申 / 乙丑 / 庚辰`
- 作者重點不是單純比高低分，而是：
  - 哪個時辰更符合星格、歸祿格與後續大運
- 目前程式：
  - `己卯時`：`+121.6`
  - `庚辰時`：`+76.9`
- 結論：
  - 目前模型只能提供「哪個候選更強、更順」
  - 尚不能代替書中那種結合身份背景的最終判定

## 對測試的落地
- 建議後續補 Rust 回歸測試：
  - 趙薇：修正時辰比書傳時辰顯著改善
  - 歐巴馬：保留三候選比較能力，用來監看 gap

## 本篇實作結論
- 第十二篇最適合做成「候選時辰比較工具」
- 不強行自動選出唯一正確時辰
- 讓量化結果成為驗證候選時辰的底稿
